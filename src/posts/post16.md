개인 프로젝트야 그냥 `main` 하나로 밀어붙여도 상관없지만,

**팀 단위 개발**로 가면 이야기가 완전 달라진다.

코드를 누가, 언제, 어떤 흐름으로 합치는지가 곧 **프로젝트 안정성**이 되거든.

그걸 정리해놓은 게 바로 **브랜치 전략(Branch Strategy)** 이다.

이번 글에서는 가장 대표적인 두 가지 —

**Git Flow**와 **Trunk Based Development**를 비교해본다.

---

## 1. Git Flow — “정석적”이지만 복잡한 전략

Git Flow는 말 그대로 **체계적인 릴리즈 프로세스**를 전제로 한다.

버전 관리, QA, hotfix 등 모든 게 분리되어 있어서 **안정적인 운영에 적합**하다.

### 기본 구조

```
main
│
├── develop
│   ├── feature/회원가입
│   ├── feature/결제
│   ├── release/v1.2.0
│   └── hotfix/v1.2.1

```

### 브랜치 역할

| 브랜치      | 역할               |
| ----------- | ------------------ |
| `main`      | 실제 배포용 브랜치 |
| `develop`   | 개발 통합 브랜치   |
| `feature/*` | 기능 개발용        |
| `release/*` | 배포 전 QA용       |
| `hotfix/*`  | 긴급 수정용        |

### 장점

-   여러 버전을 병행 관리하기 쉽다.
    (운영 중인 버전 + 개발 중인 버전)
-   QA / 테스트 프로세스가 명확하다.
-   대규모 프로젝트에 적합하다.

### 단점

-   브랜치가 너무 많아져서 관리가 복잡하다.
-   작은 팀에서는 오히려 오버엔지니어링이 된다.
-   머지(Merge) 지옥을 맛볼 수 있다.

### 쓰면 좋은 경우

-   앱, 게임, 대기업 플랫폼처럼 **릴리즈 주기가 명확한 프로젝트**
-   QA 팀이 따로 있는 환경
-   한 버전을 여러 사람이 동시에 다루는 경우

---

## 2.Trunk Based Development — “단순하고 빠른” 전략

요즘 스타트업이나 프론트엔드 팀에서 많이 쓰는 방식.

말 그대로 **하나의 trunk(main)** 중심으로 짧은 사이클로 개발하고 배포한다.

### 기본 구조

```
main
├── feature/header
├── feature/login

```

모든 기능은 `feature/*`에서 짧게 작업 후 바로 `main`에 병합한다.

(병합 시점엔 코드 리뷰나 테스트를 거치고)

### 핵심 개념

-   **작은 단위로 자주 머지**
-   **Feature Toggle(기능 플래그)** 로 미완성 기능은 숨긴다.
-   **main은 항상 배포 가능한 상태** 유지.

### 장점

-   빠르다. (CI/CD 자동화랑 잘 어울림)
-   브랜치 관리가 단순하다.
-   협업 시 충돌이 적다. (짧은 생명주기 덕분에)

### 단점

-   QA 단계가 따로 없으면, 잘못된 코드가 바로 main에 들어갈 수 있다.
-   discipline이 중요하다. (모든 개발자가 “깨끗한 main”을 유지해야 함)

### 쓰면 좋은 경우

-   **스타트업, 프론트엔드 팀, SaaS 제품**
-   **배포가 하루에도 여러 번 일어나는 프로젝트**
-   CI/CD 파이프라인이 잘 갖춰져 있을 때

---

## 3. 비교 정리

| 구분        | Git Flow                                              | Trunk Based                      |
| ----------- | ----------------------------------------------------- | -------------------------------- |
| 구조        | 여러 단계 브랜치 (main, develop, feature, release 등) | main 중심, feature 브랜치만 짧게 |
| 릴리즈 관리 | 명확한 버전 관리                                      | 지속적 배포 (CD 중심)            |
| 복잡도      | 높음                                                  | 낮음                             |
| 안정성      | 높음                                                  | main 관리에 따라 다름            |
| 적합한 환경 | 대규모, QA/테스트 중심                                | 빠른 배포, 스타트업 환경         |

---

## 그래서 실제로는?

대부분의 팀은 **두 전략을 섞어서 쓴다.**

예를 들어,

-   브랜치는 Trunk Based처럼 단순하게 쓰되,
-   릴리즈 브랜치(`release/v1.0`)만 따로 둬서 안정성 확보한다.

이런 식의 **절충형 전략**이 가장 현실적이다.

---

## 마무리

브랜치 전략은 “무조건 이게 정답”이 없다.

팀 규모, 배포 주기, 코드 리뷰 문화에 따라 완전히 달라진다.

**작은 팀 = Trunk Based**,

**큰 팀 = Git Flow**,

라고 생각하면 대체로 맞는다.

하지만 중요한 건 “전략을 정했으면 모두가 일관되게 지키는 것”.

Git 전략은 복잡한 게 아니라, **협업의 언어**라고 보는 게 맞다.
